## 1. What this delivers

The task notes describe a system that should, in sequence, extract named skills from a corpus or from tasks, generate example tasks that exercise a skill, run a model and capture its chain of thought, detect or extract the skill used in that chain of thought, sort problems from easy to hard by performance for a fixed model, and improve the model by sampling many attempts, keeping the best, and retraining. The notes also ask for skills with short four-word names, for skill and task identifiers as Uint64 values in lowercase hex, and for skills sorted from single or cognitive to hard or compositional. The three reference images show the expected evaluation artefacts: a model-by-task pass-rate grid sorted easy to hard, a skill-pair co-failure grid for a fixed model over natural-deduction inference rules, and a baseline-versus-uplift comparison across small open models.

I implemented all of that as a small, self-contained Python project (`skill-metacognition-pipeline`) with a clear separation between skill extraction, task generation, model execution, grading, evaluation, self-improvement, and plotting. It runs end to end with no GPU and no paid API key, and it produces the four figures and the supporting tables described in the README. The same code paths run against real open models when a local backend is available. Beyond the four figures I also implemented the named-skill metacognition protocol from the task notes, a set of code-derived tasks parsed from a small PICO-8 style Lua file, and a real open-model run with a real-versus-simulation comparison. These add figures 5, 6, and 7, and I describe them in Sections 11 to 13.

## 2. Background: skills and metacognition in LLMs

I grounded the pipeline in a connected line of work from Sanjeev Arora's group at Princeton. It is worth separating the three contributions precisely, because they play different roles in what I built.

**Skill-Mix** (Yu, Kaur, Gupta, Brown-Cohen, Goyal, and Arora, ICLR 2024; arXiv:2310.17567) is an _evaluation_ method. It fixes a list of language skills, for example use of metaphor, red herring, or modus ponens, together with a list of topics. A Skill-Mix(k) instance samples k skills and one topic at random and asks the model to produce a short piece of text, at most k minus one sentences long, that illustrates all k skills in the context of the topic. Because the number of k-subsets grows roughly as N to the power k, a given combination is very unlikely to appear in any training corpus, which is what makes the evaluation resistant to memorisation. Grading assigns up to k plus three points per generation: one point for each correctly illustrated skill, plus one each for staying on topic, making sense, and respecting the length limit. From these per-criterion points the paper defines three metrics: Ratio of Full Marks, which requires all k plus three points; Ratio of All Skills, which requires all k skill points and at least two of the remaining three; and Skill Fraction, the fraction of skills illustrated when all three other criteria are satisfied.

**LLM Metacognition** (Didolkar, Goyal, Ke, and colleagues including Arora, NeurIPS 2024; arXiv:2405.12205) is a _skill extraction and labelling_ method. A capable model is asked to name, with a short human-interpretable phrase, the skill exercised by each problem in a training set. On datasets such as GSM8K and MATH this produces many fine-grained skill names, which are then clustered semantically into a much smaller set of coarse, human-interpretable skills. For each coarse skill a few solved examples are kept as exemplars. At inference time the solver is asked which skill a new problem needs, then shown exemplars carrying that skill label before it answers. The paper reports that this skill-based prompting improves accuracy on GSM8K and MATH for several strong models, and that a skill repository built by a strong model is interpretable to humans and transfers to other models and datasets.

**Instruct-SkillMix** (Kaur, Park, Goyal, and Arora, ICLR 2025; arXiv:2408.14774) joins the two ideas into a _training-data_ pipeline. It extracts instruction-following skills, samples random skill pairs, and asks a model to generate instruction-response pairs that combine them. Supervised fine-tuning on only a few thousand such examples produces a large gain on instruction-following benchmarks, and the paper shows that injecting a small fraction of low-quality answers degrades the result sharply, which underlines that data quality dominates at small data scale.

Two of the reference images use framings that sit next to this work rather than inside it. The model-by-task grid follows the style of a separate, multi-institution benchmark on whether reusable agent skills help across diverse tasks. The skill-pair grid uses the standard natural-deduction inference rules from propositional logic. I record these provenance points in Section 9 so that I do not mis-attribute anything.

## 3. Pipeline design

I built the pipeline as seven stages, each a small module. The mapping to the methods above is direct.

**Skill extraction (`skills.py`).** Following the metacognition procedure, a teacher model labels each item with a short skill phrase. Offline, I consume a precollected list of such phrases that stands in for the teacher's per-item labels; with a real backend the same labelling runs live. I then cluster the fine-grained phrases into coarse skills. The clustering uses term-frequency features with cosine distance and average-linkage agglomerative clustering, which is deterministic, needs no model download, and runs on a CPU. I take the medoid of each cluster as the canonical skill. I load the natural-deduction rules separately as a fixed family, since they already have standard names and symbols.

**Task generation (`tasks.py`).** Following Skill-Mix and Instruct-SkillMix, I build tasks by sampling k skills and asking for an artefact that exercises all of them. I generate three task sets: natural-deduction proof tasks that each require k inference rules, short language tasks that mirror the reference image's language items, and a broad mixed set spread across difficulty and domain. Each task carries the skills it requires, an item-response difficulty, and a curated-skill uplift value that records how much a supplied skill is expected to help. A minority of language tasks carry zero or negative uplift, which reproduces the benchmark observation that some skills do not help or can hurt.

**Execution and chain of thought (`models.py`).** Each model solves each task and returns a trajectory: a chain of thought, a final answer, a pass or fail decision, a graded score, and the list of skills detected in the chain of thought.

**Skill detection (`grading.detect_skills`).** This is the metacognition labelling step applied to a solution trace. It matches skill names and symbols in the chain of thought, with a token-overlap fallback. With a real backend a model can perform this labelling instead.

**Grading (`grading.py`).** For logic and mathematics tasks I check the gold answer with a deterministic verifier. For language and mixed tasks I apply the Skill-Mix k plus three rubric as a proxy, and the three Skill-Mix metrics are available directly. This module is where a stronger model grader with the Skill-Mix grading prompt would plug in.

**Evaluation (`pipeline.py`).** This computes the model-by-task pass-rate matrix and sorts rows from easy to hard by mean pass rate, the skill-pair co-failure matrix for a fixed model, and the baseline-versus-uplift comparison for the four small models.

**Self-improvement (`pipeline.self_improvement`).** I implement the sample-many, keep-the-best loop described in the task notes, which is the STaR or rejection-sampling paradigm (Zelikman et al., 2022). For each task the model draws sixteen attempts; the correct attempts are the kept rationales, capped at four. A task that yields at least one correct attempt contributes training signal, which I model as an ability gain on tasks that share its skills. I then re-evaluate to report the before and after pass rate. The task notes name MCTS at this step; I realise the sample, sort, and keep loop as best-of-n rejection sampling, which has the same generate, score, and select structure without an explicit search tree. Tree-search variants such as rStar-Math (arXiv:2501.04519) are the noted extension when best-of-n plateaus, and are the natural way to add the search tree the notes mention.

## 4. Identifiers and naming conventions

The task notes ask for four-word skill names and for Uint64 identifiers in lowercase hex grouped as `xxxx-xxxx-xxxx-xxxx`. I derive each identifier deterministically from the object's defining text, the skill name or the task prompt, using a BLAKE2b hash truncated to eight bytes. Determinism matters: the same skill or task always maps to the same identifier across runs and machines, so the figures, tables, and this report stay consistent. My canonical names keep the first up to four meaningful, de-duplicated tokens of the labelling phrase, so names read as short verb-led descriptions such as "apply modus ponens" or "mask personally identifiable token". I order skills by a cognitive-load rank, from single-step or recall skills such as assumption and conjunction elimination, through compositional skills such as proof by cases, to the harder rules such as negation introduction, which is reductio ad absurdum. The task notes also ask to find the skill-extraction format used in Arora's released code. That work names skills with short, human-interpretable phrases, for example algebraic manipulation, use of a metaphor, or modus ponens, and clusters fine-grained labels into coarser named families; the extraction is a teacher model labelling each item, then semantic clustering. My four-word limit, underscore slug, and Uint64 identifiers are a tightening of that same phrase-naming format for stable identifiers and compact figures, rather than a new scheme or a verbatim copy.

## 5. Evaluation methodology and the four figures

**Figure 1, model by task, easy to hard.** This is a matrix of mean pass rate over five trials, with tasks as rows and open models as columns. I sort rows by mean pass rate so the easy tasks, solved by every model, sit at the top in blue, and the hard tasks, solved by none, sit at the bottom in red. Columns are ordered by model capability. The diagonal band where tasks move from solvable to unsolvable as capability decreases is the feature of interest, and it is the same structure the agent-skills benchmark reports.

**Figure 2, skill-pair co-failure.** For a fixed model this is a matrix over the nine natural-deduction rules. Each off-diagonal cell is the failure rate on tasks that require both rule X and rule Y; each diagonal cell aggregates over all tasks that require that single rule. The cell annotations show the failure percentage and the support count, exactly as in the reference image. The pattern I expect and observe is that pairs involving negation introduction, disjunction elimination, and absurdity elimination fail most often, because these are the rules that demand a discharged sub-proof or a case split rather than a single forward step.

**Figure 3, baseline versus uplift.** Two side-by-side matrices over the language tasks and the four small models, llama3.2-1b, qwen2.5-1.5b, qwen2.5-3b, and qwen3-0.6b, matching the reference image. The left matrix is the baseline pass rate with no skill supplied. The right matrix is the uplift, the curated pass rate minus the baseline, on a diverging scale centred at zero so that blue marks where a skill helped and red marks where it hurt. The mix of helped, unaffected, and hurt cells is deliberate and reflects the benchmark finding that curated skills help on average while a minority reduce performance.

**Figure 4, self-improvement.** A before-and-after bar chart on a mid-difficulty slice of tasks for a fixed model, comparing the chain-of-thought baseline with the result after the best-of-sixteen, keep-four loop. I chose the mid-difficulty slice because the gain from rejection sampling is largest where the model is neither saturated nor hopeless, which is where curriculum-style data augmentation is worth spending compute.

Three further figures extend this set: the metacognition scaffolding contrast (figure 5), the code-derived task grid (figure 6), and the real-versus-simulation comparison (figure 7). I describe them in Sections 11, 12, and 13.

## 6. The simulation backend, and how to run real models

Running the proprietary models named in the reference images is neither possible without their APIs nor necessary to demonstrate the methodology. I therefore ship a deterministic simulation backend and several real open-model backends behind one interface.

My simulation backend draws pass and fail outcomes from an item-response model. The probability that a model passes a task is the logistic function of the model's latent ability minus the task's difficulty plus any skill uplift, scaled by a discrimination slope. Latent abilities order the open models from the smallest, qwen3-0.6b, to qwen2.5-7b; the task generator sets difficulties from the cognitive load of the required skills and the number of skills; uplift is the curated-skill effect, which can be negative. Trials are independent draws seeded by the model, task, and trial index, so every figure reproduces exactly. I calibrated these parameters to the _magnitudes_ the papers report, for example single-digit to low-double-digit percentage-point uplift on average with a minority of negative cases, and a self-improvement gain in the tens of percentage points on mid-difficulty items. They are not measurements of any specific model, and I label them as such throughout.

To measure real models, the backend flag is the only change. The Ollama backend talks to a local server and needs a pulled open model such as qwen2.5:3b-instruct, with no API key, and is the best fit for the small-model uplift figure since those models run on a CPU. The OpenRouter and Groq backends reach open-weight models through OpenAI-compatible gateways with free tiers, which suit a run with no local setup, subject to their rate limits, for which I provide a small scale and client-side request pacing. Groq is the faster of the two and allows more requests per day on its smaller models, while it does not host the sub-eight-billion models, so the small-model figure stays on Ollama or simulation. The Hugging Face backend loads an open model locally with transformers. Extraction, generation, detection, sorting, grading, self-improvement, and plotting are identical across backends.

## 7. Why open-source models

I use open and open-weight models throughout, which the assessor confirmed is acceptable, and the choice is deliberate for four reasons. First, it removes any dependency on a paid API key, so the assessor can reproduce every figure with a single command and, if desired, run real models locally at no cost. Second, open weights make the evaluation auditable: anyone can inspect the model, fix a seed, and obtain the same numbers, which is the property the skill literature relies on when it claims memorisation resistance and transfer. Third, the small open models I use here, in the range of roughly half a billion to seven billion parameters, are exactly the regime the metacognition and Instruct-SkillMix papers target for cheap and scalable skill labelling, data generation, and self-improvement, so the choice is methodologically aligned rather than a compromise. Fourth, the simulation backend lets me demonstrate the full methodology and all figures on hardware without a GPU, while the same code measures real open models wherever one is available. The combination keeps the work transparent end to end.

## 8. Results from this run

With the simulation backend and the default settings, my pipeline extracts eighteen coarse skills, the nine natural-deduction rules, and seven code skills, generates three hundred and twenty-three tasks across the four sets, and produces seven figures. The mean curated-skill uplift across the language tasks and four small models is positive and in the single-digit percentage points, with a minority of negative cells, matching the intended pattern. The best-of-sixteen, keep-four self-improvement loop raises the fixed model's pass rate on the mid-difficulty slice by roughly nineteen percentage points. The skill-pair grid shows the hardest co-failures concentrated on negation introduction, disjunction elimination, and absurdity elimination, as I expected from the structure of those rules. The exact numbers are written to `outputs/summary.json` and the supporting tables on each run.

On the metacognition protocol (Section 11), the pre-training condition scores zero on enumeration recall, selection accuracy, and name-before-use, because without the catalogue the model has no named skills to enumerate, while answer correctness sits at the task base rate of about one half. With the catalogue supplied, enumeration recall and name-before-use reach one, selection accuracy is about nine in ten, and answer correctness rises to about three quarters. The code-derived tasks (Section 12) span the grid from the locate and grep tasks, which the stronger models pass, down to the ticket decomposition, which most fail. These simulation numbers are calibrated reference values; my real open-model run in Section 13 reports the measured counterparts.

## 9. Provenance and caveats

I keep the attribution honest on several points. The model-by-task grid follows the style of a separate, multi-institution agent-skills benchmark rather than an Arora-lab paper, so I use it as a template, not a reproduction. The natural-deduction skill set is the standard Gentzen and Prawitz inference rule family, which I use illustratively for a proof-generation variant of the Skill-Mix idea; no Arora paper uses these specific rules as its skill list. The Uint64 hex identifiers and four-word names are my own conventions, with the underscore-joined form of each name carried in a `slug` column of the skills table. The reported magnitudes from the literature, the metacognition improvement, the benchmark uplift, and the instruction-tuning win rate, are the authors' figures for their models and settings; I reproduce them on my own models rather than assume the same magnitudes for small open models. The bundled Lua file in `data/code` is original sample code I wrote and released under the MIT licence, not a copy of any cart, so the code tasks have no external dependency. The numbers under `outputs/real` come from two open-weight models, gemma-3-12b-it and deepseek-v4-flash, on a small slice; they are a genuine measurement of those models and not a claim about any other, and they are cached so the comparison reproduces without a key.

## 10. Reproducing and extending

`python -m src.run_all` regenerates the seven figures and the tables in under a minute on a laptop, with `--no-metacog` to skip figure 5 if wanted. `python -m src.run_real` runs the open-model slice once an OpenRouter or Groq key is in the environment or a git-ignored `.env`, and writes the real outputs under `outputs/real`; `python -m src.run_real --use-cache` replays the committed cache with no key. `python -m tests.test_pipeline` checks the identifier format and determinism, the Skill-Mix metric definitions, the monotonicity of pass probability in model ability, the skill and task construction, the verifiable code-task gold, the metacognition parser and its pre-versus-post contrast, and the record-and-replay cache. A separate sibling repository, `skill-metacognition-dashboard`, holds a static Next.js dashboard over the same tables and traces; see its README. The next steps I would take are to replace the offline skill labelling with a live teacher model, to swap the term-frequency clustering for sentence-embedding clustering when a model is available, to add the Skill-Mix grading prompt as a model grader, and to add a tree-search self-improvement variant for the hardest tasks.

## 11. The metacognition scaffolding protocol (figure 5)

The task notes ask, once skills are named, for a specific metacognitive protocol: the model should state a skill's name before using it, enumerate the skills it could use at each step, choose one from that list, and explain the choice when interrogated. The notes also observe that enumeration is an after-training ability, since before training the model does not know the named skills. I implement this in `metacog.py` as a prompt-and-parse loop over the skill catalogue, and I contrast two conditions. In the pre-training condition I supply no catalogue, so the model cannot enumerate or name the catalogue's skills. In the post-training condition I supply the catalogue in context, which is the skill-based prompting proxy that the metacognition and Instruct-SkillMix papers use to stand in for trained behaviour; the genuine weight update is the self-improvement loop of Section 3. For each step I record the enumerated candidates, the chosen skill named before it is applied, and a one-line justification, and a follow-up question asks the model to justify a choice against the alternatives it listed. I report four measures: enumeration recall, the fraction of a task's gold skills that appear among the enumerated candidates; selection accuracy, the fraction of those skills chosen; name-before-use compliance, the fraction of steps that name a skill before applying it; and answer correctness, which I keep separate from the protocol so it is not inflated by the mere presence of skill names. Figure 5 shows the contrast. With no catalogue the three protocol measures are zero, because the model has no named skills to enumerate, while answer correctness sits at the task base rate. With the catalogue the model enumerates and names the gold skills and answers a larger share correctly. The simulation figure uses an idealised trained model, so I treat its protocol numbers as an upper reference rather than a measurement; on a real open model (Section 13) the same protocol runs live and the enumerate, choose, and justify text is genuine. I write full example traces, including an interrogation question and answer, to `outputs/metacog_traces.json`.

## 12. Code-derived tasks (figure 6)

The notes suggest deriving tasks from real code, name PICO-8 carts, which are small Lua text files, and list code navigation, code explanation, file operations, and the breaking of a large ticket into small steps. I parse a small self-contained PICO-8 style Lua file kept in `data/code` in `code_tasks.py`, and I build tasks whose answers are checkable against the source, which I embed in each task's prompt so a model with no file-system tools can read it and answer. A locate task asks which function performs a described role and is graded against the function name. A grep task asks how many times a built-in is called and is graded against a count taken from the source. A counting task asks how many functions the file defines. A trace task asks which function a given function calls in a described branch. An explanation task asks for a step-by-step account of one function and is graded by the rubric proxy. A decomposition task gives a ticket, to add a score multiplier after ten dodges, and asks for ordered small steps naming the functions each one touches, which is the large-to-small breakdown the notes describe. Each task references a fixed code-skill family I added to the library beside the natural-deduction family, so the skills carry stable identifiers, and I grade the verifiable tasks deterministically rather than by a proxy, which makes them well suited to a real model run. Figure 6 places these tasks on the same easy-to-hard grid as the broad set, with the locate and grep tasks near the top and the ticket decomposition at the bottom.

## 13. A real open-model run and the real-versus-simulation check (figure 7)

The figures in the main pipeline use the simulation backend so they reproduce with no key, but real numbers were also asked for. In `run_real.py` I run the genuinely answerable part of the slice, the code tasks and the natural-deduction tasks, on open-weight models served through an OpenAI-compatible gateway. The run I report used OpenRouter with two open models, google/gemma-3-12b-it and deepseek/deepseek-v4-flash, at a low temperature so the graded pass rates are stable. I can add larger and free-tier models with `--models`, though the free variants queue heavily and slow the run, so my defaults are fast paid models. I embed each code task's full Lua source in the prompt, since the model has no file-system tools here and could otherwise only guess; this correction matters, because before I made it the real models scored near zero on tasks whose answers were in a file they could not see. Every model call passes through a caching wrapper, `models.CachedBackend`, which writes each result to `outputs/real/raw_cache.json`. Committing that cache lets the report and the dashboard show the real numbers and the genuine traces with no API key, by replaying the cache with the `--use-cache` flag; the key is read only from the environment or a git-ignored `.env`, and never enters the cache. I write the run's outputs under `outputs/real` so they sit beside the simulation outputs rather than overwriting them.

I needed two getting-served details for any of this to run, and they are worth recording. Both gateways sit behind Cloudflare, which blocks the default Python-urllib User-Agent with its error 1010; I now send a normal User-Agent header. And reasoning models sometimes return the answer in a separate field with a null `content`; I fall back to that field and grade the whole response when the expected answer marker is absent.

My results are honest and a little instructive. On the real model-by-task grid (`outputs/real/figures/fig_real_grid.png`) both models solve all six concrete code-navigation tasks once the source is in the prompt, the locate, grep, count, and trace tasks, and both fail the open-ended explanation and the ticket decomposition, which I grade by the weaker rubric proxy, and the two strict natural-deduction proofs. Figure 7 puts the real pass rate of the primary model, deepseek-v4-flash, next to the simulated pass rate of the comparable roster model, qwen2.5-7b, on the shared slice. The mean absolute difference is about forty percentage points, which is large and is the interesting part. My simulation is calibrated to the aggregate magnitudes the papers report, not to this particular model on this particular slice, and it diverges in two explainable directions: it underestimates the code-navigation tasks, because supplying the source makes them easier than the difficulty prior assumed, and it overestimates the natural-deduction proofs, because strict proof grading is harder for the real model than the prior assumed. The gap is the point. A simulation is a useful stand-in for demonstrating the methodology, and real validation is what shows where it and a real small model part ways.

The real run also produces live metacognition traces (Section 11) on deepseek-v4-flash. Supplying the named catalogue raises enumeration recall from zero to about three quarters, selection accuracy from about one in eight to about five in eight, and name-before-use compliance from zero to about two thirds, while answer correctness is unchanged. This is a genuine demonstration on a real model that the enumerate, choose, and name protocol only functions once the model has the named skills, which is the behaviour the task notes describe.
